## ADR-008: Node.js のパッケージマネージャー に npm を採用する

**日付:** 2026-05-05

### 背景

- Node.js のパッケージマネージャーとして何を使うか決定する必要が合った

### 検討した案

1. npm → **採用**
2. pnpm, yarn, bun ... → 非採用

### 決定内容

- mise を使って Node.js のバージョンを管理する

### 理由

- 今回の用途には npm で十分
- yarn は使う理由が薄い
- pnpm は良さそう。ただし、今の段階では npm との差を実感しにくい
- まずは npm を使って、フロントエンド開発の流れを掴むべき

## ADR-007: Node.js のバージョン管理 に mise を採用する

### 検討した案

## ADR-006: コンテンツのファイル構成として月次1ファイルを採用する

**日付:** 2026-05-05

### 背景

- コンテンツ（TIL記事）をどのような粒度でファイル分割するかを決定する必要があった
- 主な選択肢は「記事ごとに1ファイル」と「月ごとに1ファイル」の2案

### 検討した案

1. AstroPaper をそのまま使い、月次ファイルを1記事として扱う → **採用**
2. AstroPaper をカスタマイズして月次ファイルに対応させる → 非採用
3. 別の Astro テーマを探す → 非採用
4. テーマをゼロから自作する → 非採用

### 決定内容

- 月ごとに1ファイル（例：`src/content/blog/2026-05.md`）とする
- AstroPaper はそのまま使い、月次ファイルを1記事として扱う

### 決定理由

- 記事の内容が非常に短い場合もあり、記事ごと1ファイルというのは私の考える TIL に向かない
- 先述のように短文も多いため、記事ごと1ファイルとすると執筆や（Markdown ファイルとしての）閲覧が不便になる
- 「月次ファイル vs 記事ごとファイル」問題は Astro ブログテーマ全般に共通する問題であり、テーマを乗り換えても根本解決しない
- タグ・検索・ページネーション等の機能は v0.0.1 では不要と割り切れる（将来テーマを自作する際に改めて設計する）
- 案2（AstroPaperをカスタマイズ）は、どうせ将来自作するなら二度手間になる
- 案3（テーマ変更）は、調査だけで半日以上消費するリスクがある。また乗り換え先テーマがコードブロック表示の安定性などの品質面で AstroPaper を下回る可能性もある
- 案4（テーマ自作）は、今週公開の時間制約に合わない
- 月次ファイルというコンテンツ設計は、将来テーマを自作する際もそのまま引き継げる

### 影響

- AstroPaper のタグ・検索機能は月次粒度でしか機能しない。v0.0.1 では使わない前提で運用する
- 将来テーマを自作する際は、月次ファイルを前提としたカスタム Content Collections スキーマを設計すること


## ADR-005: コンテンツをコードと同一リポジトリで管理する

**日付:** 2026-05-05

### 背景

- TIL 記事（Markdown ファイル）をどこで管理するかを決定する必要があった
- 主な選択肢は「コードと同一リポジトリ」と「コンテンツ専用リポジトリに分離」の2案
- コンテンツとコードは性質の違うものなので、できれば別リポジトリで管理したい

### 決定内容

- コンテンツ（Markdown ファイル）はコードと同一リポジトリで管理する

### 決定理由

- v0.0.1 の時間制約のなかでは、リポジトリ分離による運用の複雑さを避けたい
- 記事数が少ない初期段階では、同居によるデメリット（リポジトリの肥大化など）は無視できる
- 将来、余裕ができた、またはコンテンツ量が増えた段階で分離を改めて検討する

### 影響

- 将来的にコンテンツとコードを分離リポジトリに移行する可能性がある。その際は改めて ADR を書くこと

## ADR-003: テーマに AstroPaper を採用する

**日付:** 2026-05-05

### 背景

- Astro.js でブログを構築するにあたり、テーマをゼロから自作するか既存テーマを使うかを決定する必要があった

### 検討した案

1. 既存テーマ（AstroPaper）を使う → **採用**
2. 別の既存テーマを使う → 非採用
3. テーマをゼロから自作する → 非採用

### 決定内容

- Astro.js のブログテーマとして AstroPaper を採用する

### 決定理由

- 今週公開を目指す時間制約のなかでは、既存テーマを使うのが現実的
- タグ・検索機能が標準搭載されており、TIL の逆引き用途に合っている
- スター数が多く、メンテナンスが継続されている
- シンプルなブログ用途に特化しており、余計な機能がない
- 案3（テーマ自作）は今週公開の時間制約に合わない
- テーマはいずれ自作する予定であるため、現時点では導入コストが低くシンプルな構成のものを選ぶ方針とした

### 影響

- AstroPaper を使い続ける期間は、テーマに起因する制約（月次ファイルとの相性など）を許容して運用する
- 将来テーマを自作する際は AstroPaper を破棄する前提で設計してよい


## ADR-002: フレームワークに Astro.js を採用する

**日付:** 2026-05-05

### 背景

- TIL ブログを構築するにあたり、使用するフレームワークを決定する必要があった
- このプロジェクトにはあまり時間を掛けたくないため、今週中にある程度動くところまで持っていく必要が合った

### 検討した案

1. Astro.js を採用する → **採用**
2. 自作 SSG + React.js 等で構築する → 非採用

### 決定内容

- フレームワークとして Astro.js を採用する

### 決定理由

- Markdown との相性が良い。Content Collections という機能がまさに「Markdown ファイルをコンテンツとして管理する」用途向けに設計されている
- SSG（Static Site Generation）に対応しており、TIL ブログのような更新頻度の低い静的コンテンツに適している
- デフォルトでクライアントに送る JavaScript をほぼゼロにできるため、表示が速い
- 学習コストが低い。Astro.js 未経験でも「Markdown を HTML にして表示する」程度であれば今週中に動くものが作れる
- 将来 React コンポーネントを混ぜたくなった場合も、Astro.js は Islands Architecture で対応できる
- 案2（自作 SSG）は今週公開を目指すなら明らかに時間が足りない


## ADR-001: このプロジェクト（myscraps）を開始する

**日付:** 2026-05-05

### 背景

- 開発作業中に得た知見や解決策が、時間の経過とともに記憶から薄れてしまう問題があった
- 同じ問題に再度直面したとき、以前の解決策を思い出せず再調査に時間を費やすことがあった

### 決定内容

- TIL（Today I Learned）形式の技術ミニブログ「myscraps」を開始する
- 開発で得た知見を一般化し、短くまとめて記録・公開する

### 決定理由

- **逆引き辞書として使う**：「あの解決策なんだっけ」というときに自分で見返せる場所が欲しかった
- **学習効率の向上**：知見を言語化して記事にまとめる作業自体が理解を深めるアウトプット訓練になる
- 得られた知見をそのまま記録するのではなく、文脈から剥がして一般化してから記録することで、再利用性を高める
- 非公開メモではなく公開ブログにすることで、記事の質に対する適度な緊張感が生まれる
